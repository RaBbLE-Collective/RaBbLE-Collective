---

## Executive Summary

The Grimoire is **technically rich but narratively fragmented.** The character and philosophy are locked in (`RaBbLE-Identity.md` is excellent), versioning is well-defined (`RaBbLE-Versioning.md` is rigorous), but **the roadmap is scattered across multiple documents with conflicting scopes, unclear blockers, and ambiguous episode/phase definitions.**

An agent can understand *who RaBbLE is*, but struggles to answer:
- **What is the current milestone?** (Epoch 0? Episode 1? Which member?)
- **What is blocking Epoch 0 exit?** (Different docs list different gaps)
- **Where do I start reading?** (Recommended path is unclear)
- **What's a "must fix" vs. "nice to have" vs. "future exploration"?** (Not prioritized)

## Critical Coherency Issues

### 1. **Episode vs. Phase Terminology Conflict**

| Document | Unit | Tracking Style |
|---|---|---|
| `RaBbLE-Versioning.md` | Episode, Echo, Evolution, Epoch | Collective-wide lockstep, vibe-based |
| `RaBbLE-Roadmap.md` | Phase (1-6) | Linear sequence |
| `sCoRE-Roadmap.md` | Episode, Plot A/B/C | Member-specific within Collective episodes |
| `RaBbLE-OS-Roadmap.md` | Episode, Plot A/B | Member-specific |
| `RaBbLE-Collective-Episode-1-Overview.md` | Episode | Collective scope |

**Problem:** Roadmap.md Phases (1-5a, 5b, 6) don't map cleanly to versioning's Episode/Echo model. A reader can't tell if "Phase 3 complete" means "Episode 1 is closer" or "this is side work."

**Example confusion:**
- Roadmap says Phase 3 (NeBuLA lore migration) ✓ complete
- Then Phase 4 (NeBuLA rebuild) = "NOT STARTED", Episode 1 blocker
- But Episode-1-Overview.md describes NeBuLA as "Phase 2-3 in progress, Phase 4+ pending"
- sCoRE-Roadmap says "Episode 1 Complete" with three plots already done

**Impact:** An agent can't tell the real state without reading three docs and reconciling them.

### 2. **Episode 1 Scope Is Ambiguous**

Three documents define Episode 1 differently:

| Document | Episode 1 Scope |
|---|---|
| `RaBbLE-Roadmap.md` (Epoch 0 section) | Not explicitly defined; lists exit conditions for Epoch 0 |
| `RaBbLE-Collective-Episode-1-Overview.md` | Aether + NeBuLA + World three-layer delivery (Phase 1-7) |
| `RaBbLE-Versioning.md` | "Episode 1 has not yet aired" — all work accumulates toward it; no scope defined |
| `sCoRE-Roadmap.md` | sCoRE Plot A/B/C done; Episode 1 air awaits Collective signal |

**What is Episode 1 actually?**
- Is it just the delivery of Aether/NeBuLA/World working together? (Episode-1-Overview suggests yes)
- Or is it Epoch 0 *exit* (bootstrap complete, all members registered, protocol contracts written)? (Roadmap suggests this)
- Or is it "when all members have v0.0.0.1 tag"? (Versioning implies this)

These are not the same event.

**Impact:** Members don't know when to tag `v0.0.0.1`. sCoRE is waiting for Collective signal. World/Aether/NeBuLA are unclear if they're Episode 1 or just pre-requisites.

### 9. **Narrative Tone Shifts**

The Grimoire oscillates between three voices:

1. **Character-driven narrative** (RaBbLE-Identity.md) — "RaBbLE is a Collaborative Substrate... many discordant voices..."
2. **Technical specification** (RaBbLE-Versioning.md, deployment docs) — "Event < Episode < Echo"
3. **Project management** (Roadmap.md) — "Phase 4 NOT STARTED; Blocked on decisions"

An agent reading linearly will hit identity, then versioning, then scattered roadmaps. The coherence breaks.

**What's missing:** A **connective narrative** that says:
- Here's who RaBbLE is
- Here's the structure we've built to express it
- Here's the current milestone and how it fits the long-term vision
- Here's what's next and why

## Recommendations

### Priority 1: Clarify Episode 1 Scope (Blocking)

Write a new doc: **`RaBbLE-Collective-Episode-1-Scope.md`**

Define:
- Episode 1 **is** (not what it contains, but what criteria mark it as "aired")
- Episode 1 **is not**
- Exact exit conditions (not aspirational; testable)
- Which repos must tag `v0.0.0.1`
- Which missing repos (World, NeBuLA, Aether, ScRibLE) are Episode 1 deliverables vs. pre-requisites vs. defer-to-Epoch-1
- Memory member: blocker for Episode 1 or Epoch 1? (Clarify the distinction)

**Consequence:** sCoRE, OS, NeBuLA, World, Aether all know when to tag and air.

---

### Priority 2: Unify and Prioritize the Roadmap

Consolidate `RaBbLE-Roadmap.md` from scattered Phases/Episodes into a single **Episode Sequence** view:

```
CURRENT STATE: v0.0.0.0 (Epoch 0, all members pre-Episode-1)

TOWARD EPISODE 1:
  ✓ DONE
    - sCoRE + server merged, active
    - Grimoire spells wired (setup.sh, status.sh, etc.)
    - NeBuLA lore migrated
    - Bootstrap verified
  
  ❌ BLOCKING (must finish before Episode 1 airs)
    - Create RaBbLE-NeBuLA git repo + manifest
    - Create RaBbLE-World manifest
    - Initialize RaBbLE-Aether as git repo + manifest
    - Write protocol contracts (manifest schema, health-ping)
    - Verify spells against live repos
    - Memory member: architecture doc + decision
  
  🔄 IN PROGRESS (can continue during Episode 1)
    - NeBuLA Phase 2-3 (Palette + Canvas2D)
    - Aether build setup
    - World page template (mostly done)
  
  ⏸️ DEFER (post-Episode-1)
    - NeBuLA Phase 4+ (Three.js, animation)
    - sCoRE server → task pipeline
    - ScRibLE implementation
```

**For each item, link to:** member roadmap, responsible repo, acceptance criteria.
